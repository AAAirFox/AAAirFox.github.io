C++智能指针

发表于2023-06-12|C++

RAII原则

“资源获取即初始化”原则

核心思想：资源生命周期+对象生命周期 绑定

  1. 在C++中的体现

    • 内存管理：智能指针是一个典型应用。通过使用智能指针（如std::unique_ptr、std::shared_ptr等），程序员可以确保在智能指针的生命周期结束时，其所指向的内存会被自动释放，从而避免了手动管理内存的繁琐和易错性。

    • 文件句柄和网络连接：对于文件句柄和网络连接等系统资源，也可以利用RAII原则进行管理。例如，可以创建一个封装了文件句柄的类，在类的构造函数中打开文件，在析构函数中关闭文件。这样，当对象不再需要时，文件句柄会自动被关闭，无需手动干预。

    • 异常安全：在C++中，异常处理是一个重要的方面。当发生异常时，局部对象会被自动销毁，其析构函数会被调用。通过RAII原则，可以确保在异常发生时资源得到正确释放，从而增强程序的异常安全性。

  2. 四个步骤

    • 设计一个类封装资源
    • 在构造函数中初始化
    • 在析构函数中执行销毁操作
    • 使用时定义一个类的对象

智能指针

管理指针，避免内存泄露，超出作用城，类自动析构

裸指针存在的问题？

  1. 难以区分指向的是单个对象还是一个数组；
  2. 使用完指针之后无法判断是否应该销毁指针，因为无法判断指针是否“拥有”指向的对象；
  3. 在已经确定需要销毁指针的情况下，也无法确定销毁机制，是用delete关键字删除，还是有其他特殊的销毁机制，例如通过将指针传入某个特定的销毁函数来销毁指针；
  4. 即便已经确定了销毁指针的方法，由于1的原因，仍然无法确定到底是用delete(销毁单个对象)还是delete[]销毁一个数组；
  5. 假设上述的问题都解决了，也很难保证在代码的所有路径中(分支结构，异常导致的跳转)，有且仅有一次销毁指针操作；任何一条路径遗漏都可能导致内存泄露，而销毁多次则会导致未定义行为；
  6. 理论上没有方法来分辨一个指针是否处于悬挂状态。

auto-ptr 【窃取】

不支持new[]的数组

  • 作为传入参数时：

    传入的实参获得原对象所有权，参数结束时被删除；

    引用或指计时，不知道函数内部操作→所以最好const

  • 成员函数：

    get()返回对象内存地址

    reset()重置对象

    release()返回+释放权；

主要产生的错误：p1被赋null后是空指针引用，可能导致程序错误；两指针用同一指针复制，销毁时会析构两次。【悬空指针不能再被操作，会报错】

不支持自定义删除器。

unique_ptr

可对数组使用s[0]

最好不要p2(p)，p可用作其他智能指计初始化，违背了唯一性.

  • 传参：只能引用传参，否则就是共享了内存。

  • 成员函数：

    get()返回指针地址

    reset()释放内程释放权

    release()当前ptr置空，返回的指针空间未被释放.

    swap()交换地址；

    get_deleter()返回删除器（可共用）

    move()

shared_ptr

动态数组需指定deleter

shared_ptr销毁时不顾及Weak Count，只用于检查weak_ptr是否过期。

  • 不是线程安全的
  • shared_ptr在循环引用（比如双链表）时会出现资源泄露的问题。解决办法：weak_ptr

weak_ptr

  • weak_ptr指向的shared_ptr如果析构了，weak_ptr将被赋什么值？

    任何指向该对象的weak_ptr都会变为“过期”状态。在C++中，一个过期的weak_ptr并不会自动被赋值为nullptr。但是，你可以使用weak_ptr的expired()成员函数来检查它是否过期。如果weak_ptr过期，expired()将返回true。

其他问题

  1. 智能指针是线程安全的吗？

    并不是，智能指针并不提供对多线程环境下共享资源的同步访问。

    std::shared_ptr的引用计数操作在多线程环境下可能需要额外的同步措施。这是因为引用计数的增加和减少操作可能涉及多个线程，如果没有适当的同步，就可能导致引用计数的不一致。

  2. 智能指针有没有内存泄漏的情况？

    • 循环引用：
      使用std::weak_ptr来替代std::shared_ptr在循环引用中的一方，打破强引用循环。

      重新设计数据结构或类的关系，避免不必要的循环引用。

    • 长时间持有不必要的引用：即使使用智能指针，如果长时间持有不必要的对象引用，也会占用内存。这可能会导致内存使用效率降低，尽管这不算是严格意义上的内存泄漏（因为内存最终会被释放）。

    • 异常安全：在编写处理智能指针的代码时，需要确保在抛出异常的情况下，智能指针也能正确地管理内存。否则，如果异常导致某些智能指针未被正确删除，就可能发生内存泄漏。

    • 自定义删除器：如果为智能指针提供了自定义的删除器，且删除器本身有错误，也可能导致内存泄漏。
      如何解决？

    解决方案：

    • 循环引用：当两个或多个智能指针互相持有对方的引用时，它们都不会被释放，从而导致内存泄漏。这在使用std::shared_ptr时尤其常见。为了避免这种情况，可以使用std::weak_ptr来打破循环引用。

    • 长时间持有不必要的引用：及时释放

    • 异常安全：
      编写代码时，确保在抛出异常的情况下，智能指针能正确管理其资源。

      使用RAII（资源获取即初始化）原则，确保资源的生命周期与对象的生命周期绑定。

    • 自定义删除器：
      确保为智能指针提供的自定义删除器是正确和安全的。

      如果自定义删除器涉及外部资源，确保这些资源在删除时得到正确释放。

需要注意的点

  1. 构造函数声明为explicit，可以避免隐式类型转换带来的问题。
  2. 拷贝构造函数和拷贝赋值运算符的参数都不是const的，否则会使得针对源对象(rhs)的赋值失败。
  3. 在析构函数中释放内存： delete _ptr;
  4. 需要重载指针的两个操作符.和->，以便用户能像使用裸指针那样使用智能指针。
