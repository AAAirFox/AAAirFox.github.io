EffectiveC（一）习惯C++

发表于2024-04-24|C++

01 视C++为一个语言联邦

C++是一个多重范型编程语言，一个同时支持过程形式、面向对象形式、函数形式、泛型形式、元编程形式的语言。

  • C
  • Object-Oriented C++
  • Template C++
  • STL

是四个语言组成的联邦政府，每个次语言都有自己的规约。

02 尽量以 const, enum, inline 替换 #define

尽量以编译器替换预处理器。

【疑问】：记号PIE也许从未被编译器看见，可能在编译器处理之前就被顶处理器移走了，于是记号可能没有进入记号表内，当运用此常量时显示3.1415926编译错误。

define可能存在的问题：

  1. 安全性问题
    • 文本替换导致的错误：宏定义是简单的文本替换，这可能导致一些非预期的行为。例如，如果宏的参数在替换文本中多次出现，且没有使用括号保护，那么操作符的优先级可能会导致问题。
    • 边界效应：宏定义没有作用域限制，可能导致在不同文件中定义的宏名冲突，或者在宏定义之后的代码中意外地扩展了不该扩展的文本。
    • 重复定义：如果在多个地方定义了相同的宏，可能会引发重复定义的错误。
  2. 可读性和可维护性问题
    • 缺乏类型检查：宏定义不会进行类型检查，这可能导致类型不匹配的错误，而这些错误在编译时可能不会被捕获。
    • 难以调试：由于宏是文本替换，调试时可能难以追踪到问题的源头，因为宏的展开可能发生在多个地方。
    • 不易于重构：如果宏定义被用于多处，修改宏定义可能需要搜索整个项目来确保所有使用的地方都被更新，这增加了重构的难度。
  3. 其他问题
    • 与内建函数或关键字冲突：如果宏的名称与C++的内建函数或关键字冲突，可能导致编译错误或不可预测的行为。
    • 无法携带调试信息：宏定义展开后的代码通常不包含原始宏定义的任何信息，这使得在调试时难以追踪宏的来源。

总之，使用 const、enum 和 inline 替代 #define 可以使代码更安全、更易于阅读和维护。这些特性提供了类型检查、作用域规则，并避免了宏定义可能导致的文本替换问题。

03 尽可能使用const

  1. 修饰返回值：操作符重载时

    操作符重载时，如果不限定const，(a * b) = c语句就是可行的了。但一个良好的用户自定义类型的特征是避免于内置类型不兼容（条款18），所以此语句无意义。

  2. 修饰成员函数

    为了确认该成员函数可用于const对象身上——更易读 + 可操作。有const成员函数后可用来处理取得的const对象，改善C++程序效率的一个根本办法是以pass by reference-to-const方式传递对象（条款20）。

    两个成员函数如果只是constness不同，可以被重载。

    bitwise constness & logical constness

    • bitwise constness：const成员函数不可更改任何non-static成员变量。（但如果其中成员是指针，改变指针指向的对象不会引发错误）

    • logical constness：一个const成员函数可以在检测不出的情况下修改某些数据。

    如何解决坚持bitwise constness但是实现是logical constness的情况？

    • 用mutable修饰可能改变的成员变量。

  3. 常量性转除

    现在需要定义const和non-const版本的两个同样的函数。const 版本的 operator[] 完全做了non- const 版本该做的一切，唯一的区别是返回值多了一个 const 限定符。因此如果能够安全地将返回值的 const 限定符移除，就可以使用一个函数实现两个版本的功能，避免代码的重复。

    由于const对象不能调用non-const成员，所以调用non-const operator的一定是一个non-const对象，因此，

    想让non-const operator[]调用const operator[]，不处理的话会在内部递归调用，所以将(*this)从TestBlock&类型转成const TextBlock&类型。最后从const operator[]的返回值中移除const。

    cast是糟不良实践（条款27）。

  • 声明为const可帮助编译器检测出错误用法，const可被施加于任何作用域内的对象、函数参数、函数返回类型、成员函数本体。

  • 编译器强制执行bitwise constness，但编写程序时应该使用“概念上的常量性”。

    函数中不能直接修改任何数据成员的值，包括非mutable的数据成员。

  • 当const和non-const成员函数有实质等价的实现时，可以令non-const版本调用const版本避免代码重复。

04 确定对象被使用前已被初始化

  • 对内置类型对象进行手工初始化，因为C++不保证初始化他们。

    避免出现未定义行为：如果不手工初始化内置类型的对象，它们的值是未定义的，即它们的初始值是不确定的。

  • 构造函数最好使用初始化列表，而不是在函数体内赋值。初始化列表排列顺序应该和class中的声明顺序相同。

    初始化列表中的参数初始化是在对象的构造函数开始执行之前进行的，而构造函数体中的赋值则是在对象的构造函数执行过程中的一部分。

    初始化列表可以避免先调用默认构造函数初始化成员变量，再在函数体内赋值的额外开销。

    顺序：为了维护提高可读性。

  • 为避免“跨编译单元初始化次序”的问题，用local static对象替换non-local static对象。

    当在不同的编译单元中定义了多个非局部static 对象时，它们的初始化次序是不确定的。如a.cpp中的static赋值时使用了b.cpp中的另一个static变量，此时b.cpp中的static可能还未初始化。

    将非局部static 对象替换为函数内的 local static 对象，因为 local static 对象的初始化次序是确定的，它们在第一次调用时才会被初始化。

    编译单元：产出单一目标文件的源码+头文件。多个编译单元内的nono-local static对象经由模版隐式具现化(implicit template instantiations)形成，不但不可能确定顺序，甚至往往不值得寻找“可决定正确次序”的特殊情况。
