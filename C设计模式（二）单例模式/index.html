C++设计模式（二）单例模式

发表于2023-08-24|C++

参考链接：https://blog.csdn.net/song240948380/article/details/123074885

  1. 定义

    保证一个类仅有一个实例，并提供一个访问它的全局访问点，该实例被所有程序模块共享。

  2. 使用场景

    网站计数器、应用日志程序、数据库连接池、线程池管理、全局唯一访问点、Web中配置对象的读取等。

    • 频繁实例化后又销毁的对象→提高性能
    • 经常使用的对象，但实例化耗费时间及资源→提高性能、减低资源损坏
    • 使用线程池之类的控制资源时，单例可以更便于之间通信

  3. 注意点

    • 私有化它的构造函数，以防止外界创建单例类的对象；
    • 使用类的私有静态指针变量指向类的唯一实例；
    • 使用一个公有的静态方法获取该实例。

  4. 考虑线程安全 + 内存泄漏

  5. 分类

    • 懒汉

      单例实例在第一次被使用时才进行初始化，这叫做延迟初始化。

      优点：节省内存，启动快。

      缺点：需要考虑线程安全，可能引发死锁。

    • 饿汉

      单例实例在程序运行时被立即执行初始化。

      优点：无需考虑线程安全，易实现。

      缺点：内存浪费（启动时就创建），可能造成长延迟启动（初始化过程可能耗费时间）

懒汉

基础实现

  1. 内存泄漏问题：

    Singleton 类的析构函数是私有的，这确保了外部代码不能调用它。但也意味着没有代码能够调用 delete Singleton::instance; 来释放这块内存。

    解决方法：智能指针、静态的嵌套类对象

  2. 线程不安全：

    多线程环境下，如果多个线程同时进⼊了 if (uniqueInstance == null) ，若此时还未实例化，也就是uniqueInstance == null，那么就会有多个线程执⾏uniqueInstance = new Singleton();，就会实例化多个实例。

内存泄漏

以下仅是解决内存泄漏问题的方法，仍有线程安全风险。

智能指针

  • 非可移植性
  • 性能开销
  • 不支持自定义析构行为：unique_ptr通过模版参数来确定析构函数

静态嵌套

为什么不能在~Singleton()中delete？

  1. 析构函数的调用时机：静态成员的析构函数在程序结束时调用，全局对象的析构函数在main函数返回后调用。然而，静态成员的析构函数调用顺序是未定义的，如果Singleton的静态成员变量instance的析构函数被调用，而程序中的其他全局对象或静态对象在Singleton析构之后还尝试访问Singleton实例，就会导致未定义行为（通常表现为程序崩溃）。
  2. 内存泄漏与单例的生命周期：单例模式的目的是确保一个类只有一个实例，并且这个实例贯穿整个程序的执行期间。如果Singleton的析构函数删除了instance，那么在程序的其他部分尝试获取单例实例时，将会得到一个空指针或未定义的行为，因为instance已经被删除了。
  3. 智能指针的局限性：即使使用智能指针（如std::unique_ptr或std::shared_ptr），也不能直接在Singleton的析构函数中管理instance的生命周期，因为智能指针的析构函数会在Singleton的析构函数中调用，这同样会导致在Singleton析构后尝试访问Singleton实例的问题。

线程安全

以下仅是解决线程安全问题的方法，仍有内存泄漏风险。

单锁

  • 为什么mutex需要是静态的？

    超出作用范围后自动析构。

  • 还存在什么问题？

    每次获取都会调用锁，造成了性能浪费。

双检测锁DCL

  • 解决问题：减少线程竞争，提高了性能。

    在只使用锁的情况下，每次调用 getInstance() 都会获取锁，这会造成性能开销。而双检测锁允许在没有竞争的情况下避免获取锁，从而在性能上有所提升。

  • 存在问题：半初始化

    在某些内存模型中（虽然不常见）或者是由于编译器的优化以及运行时优化等等原因，使得instance虽然已经不是nullptr但是其所指对象还没有完成构造，这种情况下，另一个线程如果调用getInstance()就有可能使用到一个不完全初始化的对象。

    即没有正确的同步，某种情况下会返回没有构造完全的对象，造成严重的错误。在C++11之前，只能插入两个内存屏障解决，11引入了memory model，提供了atomic实现内存的同步访问。

    内存屏障：

双检测锁DCL atomic+memory_order

C++11中，std::memory_order 是一个枚举类型，用于指定原子操作的内存顺序。原子操作是那些在多线程环境中可以安全地并发执行的操作，而无需额外的同步。std::memory_order 的各个枚举值决定了原子操作如何与内存交互，特别是在确保内存可见性和防止数据竞争方面。

atomic+memory_order可以一起，用于实现无锁并发编程。

  • 繁琐

静态【最优】

Meyers’ Singleton：C++11规定了local static在多线程条件下的初始化行为，要求编译器保证了内部静态变量的线程安全性。在C++11标准下，《Effective C++》提出了一种更优雅的单例模式实现，使用函数内的 local static 对象。这样，只有当第一次访问getInstance()方法时才创建实例。

C++11之后该实现是线程安全的，之前仍需加锁。

  • 为什么是线程安全的？

    明确规定了：**在多线程条件下，局部静态变量的初始化将是线程安全的**。

    所以当多个线程同时首次调用时，只会有一个线程执行初始化，其他线程会等待初始化后直接获取已初始化的对象，不会出现竞态条件，保证了线程安全性。

  • 这种方法与DCL相比，好处是什么？

    更简洁、线程安全、性能优势

  • 不会造成内存泄漏吗？

    局部静态变量的声明周期是整个程序运行期间，结束时自动销毁。

饿汉

基础实现

  1. 内存泄漏问题

    由于是静态成员变量，生命周期与程序生命周期相同，所以不会导致内存泄漏。

  2. 线程安全

    由于在程序启动时、main函数之前初始化，且是单线程初始化的，所以没有线程安全的问题。

  3. 可能返回未定义实例

    全局静态变量在不同编译单元中的初始化顺序是未定义的，那么instance和getInstance()二者的初始化顺序不确定。（例如类定义在头文件，在a.cpp中初始化了instance实例，在b.cpp中使用了Singleton::getInstance()，）

    如果在初始化前就get，可能返回未定义实例。

    C++11只解决了局部静态的初始化顺序，没有处理全局的顺序。
