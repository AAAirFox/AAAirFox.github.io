g++命令

命令使用说明

选择编译过程阶段

g++是GNU的c++编译器，在执行编译代码时，对应4个编译过程：

  1. 预处理（完成代码中的宏替换，查找相关库文件）生存**.i文件**，对应编译参数-E。

    g++ -E main.cpp

  2. 编译（将预处理后文件转换成汇编语言）**.s文件**，对应编译参数-S。

    g++ -S main.cpp //直接生成.s汇编文件

    g++ -S main.i //将预处理产生的i文件编译生成.s文件

  3. 汇编（将汇编语言文件转换成目标代码，即机器代码）.o文件，对应编译参数-c

    g++ -c main.cpp

  4. 链接（链接目标代码或静态库、动态库等[连接器ld]）生成可执行程序，对应编译参数-o

    g++ main.o -o main

其他参数

全局参数

选项 含义
-c 编译
-S 汇编
-E 预处理
-o 连接
-combine 将多个源文件一次性传递给汇编器。（可能废弃，现在自动处理）
-pipe 使用管道代替临时文件，提高编译速度，编译的不同阶段之间通过管道直接传递数据。
-pass-exit-codes 控制编译器在编译过程的各个阶段退出时所使用的退出代码。
-x language 指定后面的输入文件使用的语言。如.txt中存放的其实是c++代码。
-v 打印较多信息，显示编译器调用的程序。
–help, –target-help 显示 gcc 帮助说明。‘target-help’是显示目标机器特定的命令行选项。
–version 显示 gcc 版本号和版权信息 。
-wrapper ？好像没有

报错处理

生成目标文件

分步编译

  1. 汇编阶段：生成.0文件

    g++ -c thread.cpp -l ./

    不加-l也可以，-l主要是为了指定存储地址。

  2. 链接阶段：生成.out可执行文件

    g++ thread.o -o thread -lpthread

    如果要编译线程，一定要添加线程库。

  3. 运行

    ./thread

直接生成

g++ thread.cpp -o thread -lpthread

运行

./main

生成静态库

在使用一个静态库的时候，main文件（或任何其他使用静态库中功能的源文件）通常需要包含（#include）静态库提供的头文件。这些头文件声明了静态库中函数的原型、类、变量等，使得编译器能够了解它们的存在和如何调用它们。

  1. 编译生成.o文件（默认生成位置在当前运行目录下）

    g++ -c ./myLib/add.cpp

  2. 生成静态文件库.a文件，也可以跳过生成.o，直接生成.a文件。

    ar crv libAdd.a add.o

  3. 链接

    g++ data.cpp test.cpp libAdd.a -o main

    运行文件大小为14k

生成动态库

链接器：将多个可重定位目标文件（由编译器生成）组合成一个可执行目标文件。这个过程包括解析符号引用、合并段以及重定位符号定义等步骤。链接器确保了程序在**编译时**能够正确地找到和链接所有必要的库和符号。

动态链接器：关注于程序运行时的库文件加载和符号解析。当程序启动时，动态链接器负责将程序所需的动态链接库（如 .so 文件）加载到内存中，并解析程序对库中函数的引用。这使得程序能够在运行时动态地链接到所需的库，而无需在编译时就确定所有的依赖关系。【属于程序执行阶段】

RPATH (Run-time Search Path)：是一个嵌入在可执行文件或共享库中的字符串，它告诉动态链接器在运行时在哪里查找共享库。

**默认的系统动态链接库**的路径是由动态链接器（dynamic linker）在运行时获取的。动态链接器负责在程序执行时解析和链接动态库。在Linux系统中，动态链接器通常是ld.so或ld-linux.so。

当可执行文件试图加载一个动态库时，动态链接器会按照一组预定义的规则来查找库文件：

  1. 系统默认路径：动态链接器会在一组默认的路径中查找动态库。这些路径通常包括/lib和/usr/lib。在64位系统中，可能还会包括/lib64和/usr/lib64。
  2. 环境变量：如果设置了LD_LIBRARY_PATH环境变量，动态链接器会在这个环境变量指定的路径中查找动态库。这为用户提供了一个灵活的方式来指定额外的库搜索路径。
  3. 配置文件：系统还会读取/etc/ld.so.conf文件中列出的路径，以及/etc/ld.so.conf.d/目录下其他库路径的配置文件。这些文件包含了需要搜索的目录路径，动态链接器会读取这些文件来添加额外的库搜索路径。
  4. 缓存：这些路径的搜索信息通常会被缓存起来，以提高查找效率。这个缓存通常位于/etc/ld.so.cache。

因此，当可执行文件尝试加载一个动态库时，动态链接器会按照上述规则在系统中查找库文件。这些默认的搜索路径是硬编码在动态链接器中的，或者由系统配置文件和环境变量指定的。通过这种方式，可执行文件能够找到并加载所需的动态库，以便程序能够正常运行。

需要考虑动态链接库文件的位置，同时一个名为libAdd.so的文件可以使用-lAdd在命令行中替代。

  1. 编译生成.o文件

  2. 生成动态文件库.so文件

    g++ -fPIC -shared -o libAdd.so ./myLib/add.cpp

    【在编译时通过``-L`命令指定编译时动态链接库的位置】

    或【添加lib路径到动态链接库环境变量中】

    -shared：该选项指定生成的库为动态链接库。

    -fPIC：表示编译为位置独立的代码，不用此选项的话编译后的代码是位置相关的所以动态载入时是通过代码拷贝的方式来满足不同进程的需要，而不能达到真正代码段共享的目的。

  3. 查看动态库是否正确链接到库

    ldd main

    • 显示库文件未查找到

      【将.so放置到系统标准库下，/usr/lib 或 /usr/local/lib，然后sudo ldconfig更新】

      【添加lib路径到动态链接库环境变量中】

      export LD_LIBRARY_PATH=[/path/to/library]:$LD_LIBRARY_PATH

      【也可以在编译时通过``-Wl,-rpath`命令指定运行时动态链接库的位置，设定RPATH 或 RUNPATH】

      -Wl,-rpath,. 指定RPATH为当前目录，将其写入可执行文件，运行时查找，不适用于发布环境

    • 链接器在编译时并不关心在程序实际运行时如何找到这个库文件，这部分由动态链接器在程序启动时完成。

      -Wl,-rpath,. 是一个链接器选项，它告诉链接器在生成的可执行文件中设置一个运行时库搜索路径（RPATH）。RPATH 是嵌入在可执行文件中的，并告诉动态链接器在启动程序时首先在哪里查找动态链接库。如果不设置 RPATH 或类似的运行时搜索路径（如通过设置 LD_LIBRARY_PATH 环境变量），则动态链接器将只会在默认的系统库路径中查找所需的 .so 文件。

  4. 运行

gdb调试

在Linux环境下使用C++进行调试，通常使用GDB（GNU调试器）。以下是使用GDB进行调试的基本步骤：

  1. 编译代码时，加上 -g 选项以包含调试信息。如果不做这一步，后面没法打断点。

    g++ -g -o main main.cpp

    如果写反了，会把源代码删除

  2. 启动GDB调试程序。

    gdb my_program

    gdb main.exe --silent #关闭免责信息

  3. 在GDB中，可以使用各种命令来控制程序执行、查看变量值等。

    常用的GDB命令：

    参数列表:

    命令 命令缩写 命令说明
    list l 显示多行源代码
    info i 描述程序的状态
    run r 开始运行程序
    step s 执行下一条语句,如果该语句为函数调用,则进入函数执行其中的第一条语句。
    或step [count]
    next n 执行下一条语句,如果该语句为函数调用,不会进入函数内部执行(即不会一步步地调试函数内部语句)。
    或next [count]
    until u until #结束当前循环体代码运行
    until [location] #运行到当前行号
    start 开始执行程序,在main函数的第一条语句前面停下来
    continue c 继续程序的运行,直到遇到下一个断点
    finish 结束当前正在执行的函数，并跳出函数后暂停程序。（仅在内层函数有用）
    return 结束当前调用函数并返回指定值，到上一层函数调用处停止程序执行。
    kill k 终止正在调试的程序
    display disp 跟踪查看某个变量,每次停下来都显示它的值 undisplay #解除
    print p 打印内部变量值
    set var name=v 设置变量的值
    break b 设置断点,程序运行到断点的位置会停下来
    catch 监控某一事件的发生，等同于在程序中该事件发生的位置打普通断点。
    tcatch 仅监控第一次事件的发生。
    watch 监视变量值的变化：只有当被监控变量（表达式）的值发生改变，程序才会停止运行。
    rwatch 停止在程序中出现读取目标变量（表达式）的值的操作
    awatch 停止在程序中出现读取目标变量（表达式）的值或者改变值的操作
    condition 为三种断点设置条件中断。
    condition [break_num] [expression]
    condition [break_num] #删除
    ignore ignore [break_num] [ignore_count]
    delete 删除断点 delete [break_num]
    clear clear [location]删除指定位置处的所有断点，参数 location 通常为某一行代码的行号或者某个具体的函数名。
    backtrace bt 产看函数调用信息(堆栈)
    frame f 查看栈帧
    file 装入需要调试的程序
    quit q 退出GDB环境

  4. GDB调试多线程程序。

    使用 GDB 调试多线程程序时，默认的调试模式为：一个线程暂停运行，其它线程也随即暂停；一个线程启动运行，其它线程也随即启动。

    调试命令 功 能
    info threads 查看当前调试环境中包含多少个线程，并打印出各个线程的相关信息，包括线程编号（ID）、线程名称等。
    thread [id] 将gdb内线程编号为 id 的线程设置为当前线程。
    thread apply [id1 id2] [command] id… 表示线程的编号；command 代指 GDB 命令，如 next、continue 等。整个命令的功能是将 command 命令作用于指定编号的线程。当然，如果想将 command 命令作用于所有线程，id… 可以用 all 代替。
    thread find [thread_name] 在多线程时，找到某一线程。
    break [location] thread [id] 为特定线程设置断点：在 location 指定的位置建立普通断点，并且该断点仅用于暂停编号为 id 的线程。
    `set scheduler-locking [off on
    • off：不锁定线程，任何线程都可以随时执行；
    • on：锁定线程，只有当前线程或指定线程可以运行；
    • step：当单步执行某一线程时，其它线程不会执行，同时保证在调试过程中当前线程不会发生改变。但如果该模式下执行 continue、until、finish 命令，则其它线程也会执行，并且如果某一线程执行过程遇到断点，则 GDB 调试器会将该线程作为当前线程。

  5. GDB调试多进程程序。

    在多数 Linux 发行版系统中，GDB 并没有对多进程程序提供友好的调试功能。无论程序中调用了多少次 fork() 函数（或者 vfork() 函数），从父进程中分离出多少个子进程，GDB 默认只调试父进程，而不调试子进程。

    • 调试子进程方法：在得知目标子进程 ID 号的前提下，可以借助 attach 命令来启动 GDB 对其进行调试。

    • 如果遇到死循环，可以使用命令临时改变变量的值，从而跳出循环，执行后续代码。

    • 对于内核版本为 2.5.46 甚至更高的 Linux 发行版系统来说，可以通过修改 follow-fork-mode 或者 detach-on-fork 选项的值来调整这一默认设置。

      对于使用 fork() 或者 vfork() 函数构建的多进程程序，借助 follow-fork-mode 选项可以设定 GDB 调试父进程还是子进程。

      set follow-fork-mode [parent/child]

      set detach-on-fork [on/off]

      命令语法格式 功 能
      (gdb)show detach-on-fork 查看当前调试环境中 detach-on-fork 选项的值。
      (gdb) info inferiors 查看当前调试环境中有多少个进程。其中，进程 id 号前带有 * 号的为当前正在调试的进程。
      (gdb) inferiors id 切换到指定 ID 编号的进程对其进行调试。
      (gdb) detach inferior id 断开 GDB 与指定 id 编号进程之间的联系，使该进程可以独立运行。不过，该进程仍存在 info inferiors 打印的列表中，其 Describution 列为 ，并且借助 run 仍可以重新启用。
      (gdb) kill inferior id 断开 GDB 与指定 id 编号进程之间的联系，并中断该进程的执行。不过，该进程仍存在 info inferiors 打印的列表中，其 Describution 列为 ，并且借助 run 仍可以重新启用。
      remove-inferior id 彻底删除指令 id 编号的进程（从 info inferiors 打印的列表中消除），不过在执行此操作之前，需先使用 detach inferior id 或者 kill inferior id 命令将该进程与 GDB 分离，同时确认其不是当前进程。

  6. 浏览帧栈信息

    选项 作用
    backtrace 显示栈的所有帧。bt等同于backtrace
    frame n 设置帧n为当前帧 f n
    up 设置父帧为当前帧
    down 设置子帧为当前帧
    frame 显示当前帧
    info frame 显示当前帧的局部变量、形参等信息
    info args 显示当前帧的参数
    info locals 显示当前帧的本地变量
    info reg 显示当前帧的寄存器
